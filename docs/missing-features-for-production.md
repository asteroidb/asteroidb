# AsteroidDB: 本番アプリケーション活用に向けた不足機能の分析

## 概要

AsteroidDB は PoC としてアーキテクチャの骨格（CRDT ベースの Eventual/Certified 二層モデル、配置ポリシー、Authority 合意）がしっかり実装されている。
本ドキュメントでは、実際のアプリケーションに活用する際に不足している機能・領域を洗い出す。

---

## 1. ストレージ層の本格化

**現状**: `HashMap` ベースのインメモリストア + JSON スナップショットのファイル永続化

| 不足点 | 詳細 |
|--------|------|
| WAL (Write-Ahead Log) | クラッシュ時に最後のスナップショット以降の書き込みがすべて消失する |
| ディスクベースエンジン | データ量がメモリを超えると動作不能。LSM-tree や B-tree ベースのディスクエンジン (RocksDB, sled 等) が必要 |
| バイナリシリアライゼーション | JSON スナップショットでは大規模データで性能問題。bincode/MessagePack 等が必要 |
| 細粒度の同時アクセス制御 | `Mutex<Store>` によるグローバルロック。読み書き分離やシャーディングが必要 |

---

## 2. ネットワーク / 通信の本格化

**現状**: HTTP (reqwest + axum) ベースの同期通信

| 不足点 | 詳細 |
|--------|------|
| TLS/mTLS | ノード間通信が平文。本番では mTLS による相互認証が必須 |
| gRPC / バイナリプロトコル | JSON over HTTP は高頻度同期にはオーバーヘッドが大きい |
| コネクションプーリング / 多重化 | ノード数増加に伴うコネクション爆発の対策がない |
| バックプレッシャー制御 | 同期ストームが発生した際にノードが過負荷になる |

---

## 3. 認証・認可・セキュリティ

**現状**: 単一 Bearer トークン (`ASTEROIDB_INTERNAL_TOKEN`) によるノード間認証のみ

| 不足点 | 詳細 |
|--------|------|
| クライアント認証 | 外部 API (`/api/eventual/*`, `/api/certified/*`) は完全に無認証 |
| 認可 (Authorization) | キーレベルの ACL やロールベースアクセス制御が未実装 |
| トークンローテーション | 固定トークンのみで、漏洩時の対応ができない |
| 監査ログ | 誰が何を書き換えたか追跡できない |
| Rate limiting | DoS 攻撃への耐性がない |

---

## 4. クラスタ管理・運用

**現状**: 静的設定ファイル or 環境変数 + 簡易的な ping ベースの故障検知

| 不足点 | 詳細 |
|--------|------|
| 自動リバランス | `placement/rebalance.rs` は存在するが、実際のデータ移動ロジックがない |
| ノード追加/削除時のデータ再配置 | クラスタスケールアウト/インが安全にできない |
| 合意プロトコル | `ControlPlaneConsensus` は Authority ノードリストが静的で、Raft/Paxos 等の合意プロトコルではない |
| スプリットブレイン対策 | ネットワーク分断時の安全な動作保証が不十分 |
| ローリングアップデート戦略 | 無停止アップデートの手順が未定義 |

---

## 5. 可観測性 (Observability)

**現状**: `RuntimeMetrics` と `SloTracker` が存在し、基本的なメトリクスは収集している

| 不足点 | 詳細 |
|--------|------|
| 構造化ログの設定 | `tracing` は使っているが `tracing-subscriber` の設定がなく `println!` が多用 |
| Prometheus / OpenTelemetry 連携 | メトリクスの外部エクスポートができない |
| 分散トレーシング | リクエストのノード間追跡ができない |
| アラート連携 | SLO 違反時の通知機構がない |
| ヘルスチェックエンドポイント | Kubernetes 等の readiness/liveness probe 向けの専用エンドポイントがない |

---

## 6. クライアントライブラリ / SDK

**現状**: HTTP API のみ。CLI ツール (`src/bin/cli.rs`) は存在するが基本的

| 不足点 | 詳細 |
|--------|------|
| 公式クライアント SDK | アプリケーションから使うための Rust/Python/Go/JS のクライアントライブラリ |
| リトライ・サーキットブレーカー | クライアント側の障害対策がない |
| 一貫性レベル指定の抽象化 | Eventual vs Certified を簡単に切り替える仕組みがない |

---

## 7. データモデルの拡張

**現状**: `CrdtValue` は 4 型 (PnCounter, OrSet, OrMap, LwwRegister) のみ。値型は `String` 固定

| 不足点 | 詳細 |
|--------|------|
| バイナリ値のサポート | 画像やシリアライズ済みデータを格納できない |
| TTL / 有効期限 | キーの自動失効ができない |
| セカンダリインデックス | プレフィックス検索のみ |
| スキーマバリデーション | アプリケーションレイヤーで型安全性を保証する仕組みがない |
| 複数キー操作 | 最低限の multi-key CAS が必要な場面が多い |

---

## 8. テスト・品質

**現状**: ユニットテストは充実。`proptest` も dev-dependencies にある

| 不足点 | 詳細 |
|--------|------|
| 統合テスト / E2E テスト | 複数ノードを実際に起動してのシナリオテスト |
| カオステスト / 障害注入テスト | ネットワーク分断、ノード障害時の挙動検証 |
| 包括的ベンチマーク | スループット/レイテンシの体系的なベンチマークがない |
| ファジング | 入力バリデーションの網羅テスト |

---

## 9. デプロイ・運用基盤

| 不足点 | 詳細 |
|--------|------|
| Kubernetes マニフェスト / Helm chart | K8s 上への本番デプロイ手段がない |
| 設定管理の一元化 | 環境変数と JSON ファイルの二重管理 |
| グレースフルシャットダウンの完全性 | 進行中のリクエスト完了待ちやデータフラッシュが不十分 |
| バックアップ / リストア機能 | 定期バックアップと復旧の仕組みがない |

---

## 優先度マトリクス

| 優先度 | 項目 | 理由 |
|--------|------|------|
| **P0** | WAL + ディスクベースストレージ | データ損失防止 |
| **P0** | クライアント認証・認可 | セキュリティの基本 |
| **P0** | TLS/mTLS | 通信の暗号化 |
| **P1** | 構造化ログ + メトリクスエクスポート | 運用上の必須要件 |
| **P1** | 統合テスト | 品質保証 |
| **P1** | K8s デプロイ基盤 | 本番環境へのデプロイ |
| **P2** | クライアント SDK | 開発者体験 |
| **P2** | TTL / バイナリ値 | 実用的なデータモデル |
| **P2** | 自動リバランス | スケーラビリティ |
| **P3** | 分散トレーシング | デバッグ効率 |
| **P3** | カオステスト | 耐障害性の検証 |
