# AsteroidDB 理論的オーバーヘッド分析

通常のKey-Valueストア（Redis、etcd、DynamoDB等）と比較した場合の
AsteroidDBの理論的オーバーヘッドを各軸で分析する。

---

## 1. ストレージオーバーヘッド

### 1.1 CRDTメタデータ

通常のKVSでは値は単純なバイト列だが、AsteroidDBではすべての値がCRDTであり、
収束のためのメタデータを付随する。

| CRDT型 | メタデータ | 通常KVSとの比較 |
|--------|-----------|----------------|
| **PnCounter** | ノードごとの P/N マップ（`HashMap<NodeId, u64>` × 2） | 単一 `i64` → O(N) ノード分に膨張。N=5 で約 10× |
| **OR-Set** | 要素ごとの Dot 集合 + deferred（墓標）集合 | 単一 `HashSet<T>` → 要素あたり O(書込み回数) の Dot 蓄積 |
| **OR-Map** | OR-Set のキー管理 + LWW-Register の値 | 通常 `HashMap` → Dot + HLCタイムスタンプ分が追加 |
| **LWW-Register** | HLCタイムスタンプ（physical + logical + node_id） | 単一値 → +20〜40 bytes のタイムスタンプ |

**定量見積もり:**

- PnCounter: クラスタN台で `16N + ε` bytes のオーバーヘッド（通常は8 bytes）
- OR-Set: 要素あたり `|Dots| × (|NodeId| + 8)` bytes 追加。墓標は単調増加するため、
  削除が頻繁なワークロードではガベージコレクション（compaction）なしに無限膨張のリスクあり
- 全体的なストレージ倍率: 小さい値（カウンタ、フラグ）では **5〜20×**、
  大きい値（数KBのBlob）では **1.1〜1.5×** 程度

### 1.2 変更追跡タイムスタンプ

`Store` はキーごとに `HlcTimestamp` を保持する（デルタ同期のため）。
キーあたり追加 ~40 bytes。キーが百万件なら約40MB追加。

### 1.3 永続化フォーマット

JSON シリアライゼーション（`PersistedStore`）は、バイナリフォーマット（Protocol Buffers、
MessagePack）と比べて **2〜5×** のサイズ増加。これはMVPの選択であり、
最適化すべき最初のポイントである。

## 2. 書き込みレイテンシ

### 2.1 Eventual Write

```
通常KVS:   ローカル書込み ≈ O(1) ハッシュテーブル操作
AsteroidDB: HLC生成 + CRDTマージ + タイムスタンプ記録
```

- **HLC.now()**: システムコール (`SystemTime::now()`) + 比較・更新。~100ns 追加
- **CRDTマージ**: 型により異なる
  - PnCounter: O(N) — ノード数分のmax演算
  - OR-Set add: O(1) — 新しいDot生成のみ
  - OR-Set merge: O(|elements| × |dots|) — 全ドットの突合
  - LWW-Register: O(1) — タイムスタンプ比較のみ
- **追加オーバーヘッド**: ~1〜10μs（ローカル書込みのみの場合）

**結論:** Eventual write のローカル部分は **通常KVSとほぼ同等**。
マイクロ秒単位のオーバーヘッドは実用上無視できる。

### 2.2 Certified Write

```
通常の強一貫性DB (etcd/Raft):  リーダー → 過半数ACK → コミット
AsteroidDB:                     ローカル書込み → Authority過半数のフロンティア前進を待機
```

- **最良ケース**: 1 RTT（Authority全ノードが高速応答）
- **想定ユースケース**: 地球-宇宙間通信で RTT = 数秒〜数十分
- AsteroidDB の certified write は Authority ノードの `ack_frontier` が
  書込みタイムスタンプを超えるのを待つ。これは **Raft のログ複製と本質的に同等**だが、
  AsteroidDB は Authority が非同期にフロンティアを進めるため、
  書込みがバッチ化される可能性がある
- **最悪ケース**: ネットワーク分断時にタイムアウトまで待機（デフォルト30秒）、
  `on_timeout=Pending` 設定なら非同期に格下げ可能

**結論:** Certified write のレイテンシは Raft ベースのDBと **同オーダー（O(RTT)）**。
ただし、通常のRaftと異なりリーダー選出が不要なため、リーダー障害時のフェイルオーバー
コストがない。

## 3. 読み取りレイテンシ

### 3.1 Eventual Read

```
通常KVS:     HashMap.get(key) → O(1)
AsteroidDB:  HashMap.get(key) → O(1)
```

**オーバーヘッド: ゼロ。** 完全にローカル操作。

### 3.2 Certified Read

```
通常の強一貫性DB: リーダーに転送 or リースベースのローカル読み取り
AsteroidDB:       ローカル読み取り + ProofBundle構築 + フロンティア検証
```

- ProofBundle の構築は `AckFrontierSet` の検索 + Ed25519署名の検証
- Ed25519 verify: ~70μs/検証 × Authority数（通常3〜5）
- **合計追加レイテンシ**: ~200〜500μs

**結論:** 通常の linearizable read（リーダーRTTが必要）と比較して、
AsteroidDB の certified read は **レイテンシ面で有利**。
署名検証のCPUコストと引き換えにネットワークRTTを節約できる。

## 4. ネットワークオーバーヘッド

### 4.1 レプリケーション帯域

| 方式 | 帯域使用量 | AsteroidDB |
|------|-----------|------------|
| Raft ログ | コマンドサイズ × レプリカ数 | — |
| State-based CRDT (フル同期) | 全状態 × ピア数 | フォールバック時のみ |
| Delta-state CRDT | 差分のみ × ピア数 | **通常のパス** |

- デルタ同期: 変更があったエントリのみ送信（フロンティアベース）
- ベンチマークによると、1000エントリ中10件変更 → **約10× のペイロード削減**
- ただし、CRDT値自体がメタデータを含むため、同じキー・同じ値でも
  Raft のコマンドログより **2〜5×** 大きいペイロードになる

### 4.2 バックグラウンドトラフィック

通常のKVSにはないAsteroidDB固有のトラフィック:

1. **Anti-entropy sync** (定期的): ピアごとにデルタ同期リクエスト
2. **Membership gossip**: ピアリストの交換
3. **Authority frontier reporting**: フロンティア前進の通知
4. **Compaction digest**: チェックポイントのハッシュ検証

N ノードクラスタでの追加メッセージ数: **O(N²)** / 同期間隔
（各ノードが全ピアと同期するため）

## 5. CPU オーバーヘッド

### 5.1 CRDT マージ計算

| 操作 | 計算量 | 備考 |
|------|--------|------|
| PnCounter merge | O(N) | ノード数に比例 |
| OR-Set merge | O(\|E\| × \|D\|) | 要素数 × 平均ドット数 |
| OR-Map merge | O(\|K\| × (merge_cost)) | キー数 × 値のマージコスト |
| LWW-Register merge | O(1) | タイムスタンプ比較のみ |

通常のKVSでは書込みは単純な上書き（O(1)）なので、
**マージ操作はCRDT固有のCPUコスト**である。
ただし同期時にのみ発生し、ローカル操作では無視できる。

### 5.2 暗号処理

- **Ed25519署名**: ~50μs/署名（Authority ノードのみ）
- **Ed25519検証**: ~70μs/検証（certified read/write 時）
- **将来のBLS閾値署名**: ~1.5ms/署名、~1ms/検証

通常のKVSではTLS以外の暗号処理はないため、これは純粋な追加コスト。
ただし、certified操作でのみ発生するため、eventual-only ワークロードでは影響ゼロ。

### 5.3 HLC管理

HLC.now() と HLC.update() はいずれも O(1) でシステムコール1回。
実質的なオーバーヘッドなし。

## 6. 収束遅延（Convergence Delay）

通常のKVSにはない概念。AsteroidDB固有の「オーバーヘッド」として考慮が必要。

```
書込み → ローカル反映（即座）→ ピアへ伝播（同期間隔に依存）→ 全レプリカ収束
```

- **最良ケース**: 1同期間隔 = 数百ms〜数秒（LAN）
- **想定ケース**: 地球-宇宙間 = 数秒〜数十分
- **最悪ケース**: ネットワーク分断中は収束しない（分断解消まで）

通常の強一貫性DBでは、write 完了 = 全レプリカ反映保証。
AsteroidDB の eventual write はこの保証を提供しない。
これはオーバーヘッドというより**設計上のトレードオフ**。

---

## 7. 総合評価

### オーバーヘッドが大きい領域

| 領域 | 倍率 | 影響度 |
|------|------|--------|
| ストレージ（小さい値） | 5〜20× | 高 |
| OR-Set 墓標蓄積 | 無限増加リスク | 高（compaction必須） |
| JSON永続化 | 2〜5× | 中（最適化容易） |
| ネットワーク（同期メッセージ） | O(N²) | 中（大規模クラスタで顕在化） |
| 暗号処理（certified操作） | +200〜500μs | 低〜中 |

### オーバーヘッドが小さいまたは有利な領域

| 領域 | 評価 | 理由 |
|------|------|------|
| Eventual write レイテンシ | ≈ 同等 | ローカル操作、追加は μs 単位 |
| Eventual read レイテンシ | = 同等 | 純粋なローカルHashMap参照 |
| Certified read レイテンシ | 有利 | Raft リーダーRTT不要 |
| パーティション耐性 | 大幅有利 | CRDT により書込み継続可能 |
| リーダー障害時の可用性 | 大幅有利 | リーダーレスアーキテクチャ |

### 結論

AsteroidDBの主なオーバーヘッドは **ストレージ効率** と **CRDT メタデータの管理コスト** に
集中する。一方、レイテンシ面では eventual 操作はほぼゼロコスト、certified 操作も
Raft ベースの DB と同等かそれ以上の性能を達成可能。

このトレードオフは、AsteroidDB の想定ユースケース（高レイテンシ・パーティション頻発
環境）では **非常に合理的** である。従来のDBでは書込みが完全にブロックされる分断シナリオ
でも、AsteroidDB は eventual write を継続でき、後でCRDTマージにより自動収束する。

**最適化の優先順位:**
1. JSON → バイナリ永続化（即効性が高い）
2. OR-Set の墓標GC / compaction の積極化
3. 大規模クラスタ向けの gossip 最適化（O(N²) → O(N log N)）
